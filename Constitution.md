# Constitution - インフラ業務 仕様駆動開発 原則決定書

> この文書はプロジェクトの存在意義、基本原則、スコープと責任範囲、制約条件、ガバナンスを定義する最上位文書です。
> 事業環境の根本的な変化や法制度の改訂といった特別なことが起きない限り更新されません。

> 注記: 実装フェーズなどの可変な進捗情報は本書の対象外とし、README と CHANGELOG で管理します。

## 1. プロジェクトの存在意義

情報システム部のインフラ業務を**仕様駆動開発（SDD: Specification-Driven Development）**の手法で管理し、Microsoft Azure および生成AIソリューションを活用して、合理的・正確・スピーディな業務遂行を実現する。

## 2. 基本原則

### 2.1 仕様は生きたドキュメント

- 常に最新のプロジェクト状況を反映する動的な成果物として扱う
- 仕様はマークダウンで記述する
- Git/GitHub を使って仕様を管理し、全ての変更履歴を残す
- 仕様の更新を日常的な活動の一部とする
- 仕様の更新にかかる手間をAI活用で削減する

### 2.2 仕様は信頼できる唯一の情報源

- 知識の属人化を防ぐため、仕様はマークダウンに必ず反映させる
- GitHub 上での仕様へのアクセス権はチーム全員に開放する
- 議論が平行線になる場合は仕様を確認する習慣をつける
- 新メンバーのオンボーディングは GitHub 上のマークダウンを読むことから始める

### 2.3 仕様は変更と反復が前提

- 仕様は進化するものとして変更を前提とする
- GitHub のコミット履歴で変更履歴を差分レポートとして可視化する
- 反復サイクルを明確にし、各サイクルで仕様を見直す機会を設ける
- 変更を失敗ではなく学びの反映として履歴に残す文化を育てる

### 2.4 AIでコストを抑える

- 仕様はAIが理解できるマークダウンで記載する
- 設計案やコードを生成させる際、常に仕様を入力として与える
- 出力は鵜呑みにせず、人間のレビュー（Human in the Loop）を徹底する
- AI はアシスタントとして位置づけ、最終判断は人間が行う

## 4. 制約条件

- すべての仕様はマークダウン形式で管理する
- すべての変更は GitHub のコミット履歴として記録する
- AI生成物は必ず人間がレビューする（Human in the Loop）
- 各カテゴリの実装工程で作成されるアウトプットは、他のカテゴリのフォルダには絶対に配置せず、実装工程の処理を実施した同一カテゴリのフォルダ内に配置すること
- 新規リソースの展開はすべて Bicep で記述する
- Office系ファイル（pptx, docx, xlsx）は非構造化データの入口として扱い、マークダウンへ変換する

## 5. ガバナンス

- 本文書の変更には、プロジェクトオーナーおよび主要ステークホルダーの承認が必要
- 各カテゴリの仕様変更は本文書に定める原則に準拠すること
- 仕様変更のレビューサイクルはスプリント単位で実施する
- GitHub Actions を活用した自動テスト・検証を各カテゴリに適用する

## 6. SDDの7つの工程

本プロジェクトでは以下の工程を各カテゴリで適用する：

| 工程 | 名称 | 概要 |
|------|------|------|
| 1 | 原則決定 (Constitution) | 本文書。プロジェクトの基本原則を定義 |
| 2 | 企画・要件定義 (Specify) | 「何を」「なぜ」を定義。マークダウンで記載 |
| 3 | 設計計画 (Plan) | 技術スタック、アーキテクチャ、制約条件の整理 |
| 4 | タスク分割 (Tasks) | レビュー30分以内の粒度でタスクを分割 |
| 5 | 実装 (Implement) | タスク毎にコード記載、人間がレビュー・検証 |
| 6 | 検証・受入 (Verify/Accept) | 仕様充足の確認、自動テスト、差分レポート |
| 7 | 移行・運用 (Migration/Operation) | 本番移行と継続的フィードバックサイクル |

### 6.1 Specify と Plan における What / Why / How の役割分担

各カテゴリの実装において、以下の原則を厳密に適用する絶対仕様とする：

- **Specify（企画・要件定義）工程**: 「**What**（何を）」「**Why**（なぜ）」を定義する
  - 業務要件、解決すべき課題、実現したい状態を明確に記載する
  - マークダウンで記載し、GitHub で管理する
  - How（どのように実装するか）の記載は行わない
  
- **Plan（設計計画）工程**: 「**How**（どのように）」を定義する
  - Specify で定義された What/Why を前提に、技術スタック、アーキテクチャ、実装方針、制約条件を整理する
  - What/Why が明確であれば、複数の How の選択肢を検討し、最適なアプローチを設計する
  - ADR（Architecture Decision Records）をマークダウンで必ず作成し、主要な技術判断を記録する
  - 各ADRには最低限、以下を明記する
    - なぜその技術を選定したか（採用理由）
    - なぜその実装方式・運用方式を採用したか（方式選定理由）
    - 代替案と不採用理由

この役割分担により、要件の変更が生じた場合でも、Specify と Plan の変更影響を分離し、管理効率を高める。
